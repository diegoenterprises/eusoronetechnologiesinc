## ISSUE

`fleet.ts` has ZERO `requireAccess()` calls. Any authenticated user can view, create, update, or delete ANY company's vehicles, geofences, and fleet data. Multi-tenant data isolation is completely broken for fleet management.

All other critical routers (loads.ts, wallet.ts, billing.ts) have proper RBAC guards — fleet.ts is the outlier.

## AFFECTED USER TYPES

Catalyst/Carrier (fleet owners), Driver, Admin, Compliance Officer

## AFFECTED WORKFLOW

Fleet Management: View vehicles, manage geofences, update fleet status

## FILE(S) TO MODIFY

`frontend/server/routers/fleet.ts` — ALL procedures

## REQUIRED CHANGE

Add `requireAccess()` and `ownershipFilter()` to every fleet procedure:

```typescript
import { requireAccess } from '../services/security/rbac/access-check';
import { ownershipFilter } from '../services/security/isolation/ownership-verifier';

// For READ procedures (getVehicles, getGeofences, getGeofenceStats):
getVehicles: protectedProcedure
  .input(z.object({ companyId: z.number().optional() }))
  .query(async ({ ctx, input }) => {
    await requireAccess({
      userId: ctx.user.id,
      role: ctx.user.role,
      action: 'READ',
      resource: 'fleet',
      targetCompanyId: input.companyId || ctx.user.companyId,
    });

    const db = await getDb();
    const companyId = input.companyId || ctx.user.companyId;

    // Apply ownership filter — users can only see their own company's fleet
    const results = await db.select().from(vehicles)
      .where(eq(vehicles.companyId, companyId));

    return results;
  }),

// For WRITE procedures (createVehicle, updateVehicle, deleteVehicle):
createVehicle: protectedProcedure
  .input(z.object({ /* ... */ }))
  .mutation(async ({ ctx, input }) => {
    await requireAccess({
      userId: ctx.user.id,
      role: ctx.user.role,
      action: 'CREATE',
      resource: 'fleet',
      targetCompanyId: ctx.user.companyId,
    });
    // ... existing create logic, but set companyId = ctx.user.companyId
  }),

// For DELETE procedures:
deleteGeofence: protectedProcedure
  .input(z.object({ id: z.number() }))
  .mutation(async ({ ctx, input }) => {
    // First verify the geofence belongs to user's company
    const [geofence] = await db.select().from(geofences)
      .where(eq(geofences.id, input.id));

    await requireAccess({
      userId: ctx.user.id,
      role: ctx.user.role,
      action: 'DELETE',
      resource: 'fleet',
      targetCompanyId: geofence?.companyId,
    });
    // ... existing delete logic
  }),
```

Apply this pattern to ALL fleet.ts procedures. The key principle: users can only access their own company's fleet data. Admins and Super Admins can access any company.

## VERIFICATION

1. Login as Company A catalyst → query fleet → only see Company A vehicles
2. Try to access Company B fleet data via direct API call → 403 FORBIDDEN
3. Login as Admin → can see all companies' fleet data
4. Create vehicle → companyId automatically set to user's company
5. Try to delete another company's geofence → 403 FORBIDDEN
