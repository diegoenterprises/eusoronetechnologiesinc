## ISSUE

Multiple data integrity problems in the loads table:
1. Load 11 (LD-260301-MAIY7HZB) has deliveryDate (2026-02-02) BEFORE pickupDate (2026-03-02)
2. Loads 1-6 have lat=0, lng=0 for pickup/delivery locations (geocoding was broken)
3. Loads 1-5 have NULL distance
4. Inconsistent load number format (old: LOAD-timestamp, new: LD-date-hash)

## REQUIRED CHANGE

### Fix 1: Load 11 delivery date
```sql
UPDATE loads SET deliveryDate = '2026-03-03 00:00:00'
WHERE loadNumber = 'LD-260301-MAIY7HZB';
```

### Fix 2: Add validation to loads.create to prevent this
In `frontend/server/routers/loads.ts`, in the create procedure:
```typescript
if (input.deliveryDate && input.pickupDate) {
  if (new Date(input.deliveryDate) < new Date(input.pickupDate)) {
    throw new TRPCError({
      code: 'BAD_REQUEST',
      message: 'Delivery date cannot be before pickup date.',
    });
  }
}
```

### Fix 3: Backfill coordinates for loads with 0,0
For loads 1-6, geocode their city/state addresses. Example updates:
```sql
-- Houston, TX → Austin, TX loads (adjust based on actual addresses)
UPDATE loads SET
  pickupLocation = JSON_SET(pickupLocation, '$.lat', 29.7604, '$.lng', -95.3698),
  deliveryLocation = JSON_SET(deliveryLocation, '$.lat', 30.2672, '$.lng', -97.7431)
WHERE id IN (1, 2, 3, 4, 5, 6)
AND JSON_EXTRACT(pickupLocation, '$.lat') = 0;
```

Better approach: Use a geocoding service (Google Maps, Mapbox) to geocode the stored city/state for each load individually.

### Fix 4: Calculate missing distances
For loads with NULL distance, calculate using haversine formula:
```sql
-- Example for a Houston→Austin load (~165 miles)
UPDATE loads SET distance = 165.00, distanceUnit = 'miles'
WHERE distance IS NULL AND id IN (1, 2, 3, 4, 5);
```

Better approach: Call the distance calculation service that newer loads use.
