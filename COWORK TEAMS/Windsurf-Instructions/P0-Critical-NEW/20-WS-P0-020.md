# WS-P0-020: Double-Brokering Prevention

**Priority:** P0-CRITICAL
**Team:** Delta (Compliance & Regulatory)
**Estimated Complexity:** Medium
**Unlocks:** Broker compliance, shipper protection, regulatory trust

## Problem
Zero code exists to prevent double-brokering. A broker can accept a load from another broker and re-post it as their own with no validation. The `brokers.ts` router sets `shipperId: userId` (broker as shipper) with no origin check. The authority.ts router tracks lease relationships but doesn't use them for anti-double-brokering. Sales materials promise "double-brokering protection" to shippers and brokers.

## What to Implement

### 1. Add `originalShipperId` field to loads table
```typescript
// In drizzle/schema.ts, loads table definition:
originalShipperId: int("originalShipperId"), // Tracks the actual shipper, null = direct shipper post
brokerChainDepth: int("brokerChainDepth").default(0), // 0 = shipper direct, 1 = one broker
```

### 2. Anti-double-brokering validation in load creation
```typescript
// In loads.ts create mutation and brokers.ts create mutation:

// When a BROKER creates a load, check if they're re-brokering
if (ctx.user.role === 'BROKER') {
  // Check if this load matches an existing load the broker accepted
  const existingLoad = await db.select()
    .from(loads)
    .where(and(
      eq(loads.origin, input.origin),
      eq(loads.destination, input.destination),
      eq(loads.catalystId, ctx.user.id), // Broker was assigned as carrier
      inArray(loads.status, ['assigned', 'accepted', 'confirmed']),
    ))
    .limit(1);

  if (existingLoad.length) {
    // This broker accepted a load and is now re-posting it
    if (existingLoad[0].brokerChainDepth >= 1) {
      throw new TRPCError({
        code: 'FORBIDDEN',
        message: 'Double-brokering detected: This load has already been brokered. Re-brokering is prohibited.',
      });
    }
  }

  // Set the chain metadata
  newLoadValues.originalShipperId = input.originalShipperId || ctx.user.id;
  newLoadValues.brokerChainDepth = (existingLoad[0]?.brokerChainDepth || 0) + 1;
}
```

### 3. Pattern detection service
```typescript
// frontend/server/services/compliance/doubleBrokerDetection.ts

export async function detectSuspiciousPatterns(loadId: number) {
  const load = await db.select().from(loads).where(eq(loads.id, loadId)).limit(1);
  if (!load.length) return { suspicious: false };

  // Check for same origin/dest posted by different broker within 24h
  const similar = await db.select()
    .from(loads)
    .where(and(
      eq(loads.origin, load[0].origin),
      eq(loads.destination, load[0].destination),
      ne(loads.shipperId, load[0].shipperId),
      gte(loads.createdAt, new Date(Date.now() - 24 * 60 * 60 * 1000)),
    ));

  if (similar.length > 0) {
    // Flag for compliance review
    await db.insert(complianceAlerts).values({
      alertType: 'potential_double_brokering',
      loadId,
      relatedLoadIds: similar.map(s => s.id),
      severity: 'high',
      message: `Load matches ${similar.length} similar load(s) posted by different brokers in last 24h`,
    });
    return { suspicious: true, matches: similar.length };
  }

  return { suspicious: false };
}
```

### 4. Compliance alert for officers
Wire alerts into the notification system so COMPLIANCE_OFFICER role users get real-time alerts for suspected double-brokering.
