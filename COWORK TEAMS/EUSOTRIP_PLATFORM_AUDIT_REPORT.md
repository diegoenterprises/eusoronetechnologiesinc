## TEAM EPSILON: Financial Transaction Flows

### 4 CRITICAL ISSUES

**1. Stripe Webhook Signature Validation BROKEN**
- **File:** `frontend/server/_core/stripe.ts` line 270
- **Function:** `verifyWebhookSignature()`
- **Bug:** Function ALWAYS returns `true` regardless of input
- **Impact:** ANY attacker can send fake Stripe webhook events. They could:
  - Mark payments as succeeded without actual payment
  - Trigger fake refunds
  - Inject fraudulent settlement data
- **Fix:** Implement proper HMAC-SHA256 signature verification using Stripe's webhook secret

**2. Settlements Table Does Not Exist**
- **File:** `frontend/server/routers/settlementBatching.ts` line 55
- **Bug:** Queries a `settlements` table that is not defined in the schema
- **Impact:** ANY call to settlement batching will CRASH with a database error
- **Fix:** Add `settlements` table to schema.ts or map to the correct existing table

**3. Payments Marked Succeeded Without Validation**
- **File:** `frontend/server/routers/payments.ts` lines 137-139
- **Bug:** `createPayment()` immediately marks payment status as "succeeded" without calling Stripe API or validating funds
- **Impact:** Enables fraudulent transactions — payment records show "succeeded" with no actual money movement
- **Fix:** Integrate Stripe PaymentIntent creation and confirm actual charge before status update

**4. Platform Fees Calculated But Never Collected**
- **File:** `frontend/server/routers/billing.ts` lines 458-500
- **Bug:** Fee calculation logic exists but the collection/charge mechanism is incomplete. `getAccessorialCharges` returns empty array.
- **Impact:** Platform generates no revenue from fees despite having fee configuration UI
- **Fix:** Wire fee collection to Stripe Connect application fees on each transaction
